# 批量图片工作台交互重构方案（给 Claude Code）

## 背景

当前工作台已经具备复杂产品图生成、分镜图生成、脚本生成、视频生成等能力，但入口和页面组织比较混乱：

- 供应商页里有“设为唯一启用”，用户理解成本高，且和真实需求不匹配。
- 新建项目页承担了太多任务：供应商、模型、场景图、分镜图、生成提示词、卖点脚本信息都混在一起。
- 项目详情页缺少清晰的生产顺序，用户不知道自己处在流程哪一步。
- “场景图 B”“分镜重做”等命名偏内部，不够像用户任务。

本次目标是重排交互，不重写核心生成能力。最终体验应变成：先创建项目壳，再在项目工作台里按开放式步骤完成生产。

## 总目标

把复杂产品项目改成一个开放式 4 板块工作台：

1. 新场景图生成
2. 分镜生成
3. 脚本生成
4. 视频生成

这些板块只是推荐顺序，不是强制向导。不要把后续板块灰掉或锁死。用户应该能单独进入任何板块操作；如果缺少材料，在板块内部用提示说明“建议先准备什么”，并提供快捷入口或明确下一步。

## 不在本次范围

- 不重写图片生成队列、视频队列、provider adapter。
- 不做新的视觉品牌设计。
- 不删除已有历史数据。
- 不强制迁移旧项目为新数据结构，旧项目应尽量还能打开。
- 不把 `legacy_batch_edit` 模式彻底移除；如果保留该入口，需保持可用。

## 术语统一

需要统一替换和避免混用：

- “场景图 B”对用户改叫“新场景图”。
- “场景图 B 生成”改叫“新场景图生成”。
- “分镜重做”改叫“分镜生成”。
- “产品卖点”归入“脚本生成”板块，不放在新建项目页。
- “视频生成”单独成板块，不和脚本生成混在一个区块。

## 供应商页调整

文件重点：

- `app/settings/page.tsx`
- `app/api/providers/[id]/activate-only/route.ts` 可保留不用，除非确定无风险再删除。

交互要求：

- 删除“设为唯一启用”按钮和相关说明。
- 保留“启用 / 禁用”“编辑”“删除”。
- 页面说明改为：可以保存多个供应商，只有启用且已配置 Key 的供应商会出现在新建项目或生成设置中。
- 不要在 UI 上暗示“只能启用一个供应商”。
- `handleActivateOnly` 等前端逻辑可以移除。

验收：

- 每个供应商卡片只有启用/禁用、编辑、删除等基础操作。
- 多个供应商可以同时启用。
- 新建项目供应商选择仍只展示可用供应商，或展示不可用供应商但不可选并解释原因。

## 新建项目页调整

文件重点：

- `app/projects/new/page.tsx`
- `app/api/projects/route.ts`
- 可能涉及 `lib/db.ts`

目标：新建项目页只负责创建项目壳，不再做完整生产。

保留字段：

- 项目名称，必填。
- 产品名称，可选但建议保留。
- 产品编号，可选。
- 产品类目，可选。
- 工作流类型：复杂产品项目 / 批量编辑项目。复杂产品项目作为默认。
- 供应商选择，必填。
- 模型、画面比例、清晰度、质量、超时等基础参数。
- 图片预处理参数可以保留，但可以折叠到“高级设置”。

移出新建项目页：

- 场景图 A 上传。
- 新场景图生成提示词。
- 原始分镜图上传。
- 分镜生成提示词。
- 目标人群、语气、平台、卖点。
- 创建后自动开始跑队列。

复杂产品项目创建行为：

- 点击“创建项目”后，只创建项目记录并跳转项目详情页。
- 不要求 `sceneSeedImageId`。
- 不要求 `shotImageIds`。
- 不自动创建 scene jobs。
- 不自动创建 shot set。
- 不自动调用 `/api/projects/[id]/run`。

后端必须同步调整：

当前 `app/api/projects/route.ts` 的 complex workflow 会强制校验：

- `sceneSeedImageId`
- `scenePrompt`
- `shotImageIds`

这些校验需要放宽。建议逻辑：

- 如果请求里没有 `sceneSeedImageId` 和 `shotImageIds`，只创建项目壳。
- 如果请求里传了旧格式完整素材，可以继续兼容旧的一次性创建逻辑，但新 UI 不再依赖它。
- `projects.prompt` 可以存默认新场景图提示词或空字符串，避免 NOT NULL 报错。
- `projects.scenePrompt`、`projects.shotPrompt` 写入默认模板，供详情页使用。

验收：

- 用户只填项目名 + 供应商 + 基础参数即可创建复杂产品项目。
- 创建后进入 `/projects/[id]`。
- 新项目没有图片和任务时，详情页仍能正常展示 4 个板块。

## 项目详情页：整体结构

文件重点：

- `app/projects/[id]/page.tsx`
- `components/SceneReferencePanel.tsx`
- `components/ShotSetPanel.tsx`
- `components/ScriptPanel.tsx`
- `components/VideoGenerationPanel.tsx`
- `components/ImagePickerGrid.tsx`
- `components/ResultGallery.tsx`
- `components/JobQueueTable.tsx`

目标：详情页变成开放式步骤工作台。

顶部建议结构：

- 返回项目列表。
- 项目标题、状态、供应商、模型、任务统计。
- 操作区：开始/暂停/继续/取消队列、下载、导出、刷新。
- 下方放 4 个锚点或 tabs：
  - 1 新场景图生成
  - 2 分镜生成
  - 3 脚本生成
  - 4 视频生成

要求：

- 4 个板块都应可见或可通过 tab 切换进入。
- 不要因为缺材料而灰掉整个板块。
- 缺材料时在板块内部显示轻提示，例如“还没有新场景图，可以先上传场景图 A 生成，也可以直接上传已有新场景图。”
- 用户可以只做脚本生成、只做视频生成、只上传分镜图。

## 板块 1：新场景图生成

职责：

- 上传或选择原始场景图 A。
- 编辑新场景图生成提示词。
- 设置生成数量、并发、质量等必要参数。
- 创建并运行新场景图候选任务。
- 从生成结果中选择一张，设为“新场景图”或“场景参考图”。
- 允许直接上传已有新场景图作为场景参考。

命名：

- 标题：新场景图生成
- 子区块：原始场景图、生成设置、候选结果、当前选中的新场景图

交互提示：

- 如果没有候选图：提示“上传原始场景图后，可以生成多个新场景候选。”
- 如果已有候选图但未选择：提示“请选择一张作为后续分镜生成的场景参考。”
- 如果已有 active scene reference：显示当前选中项和更换入口。

实现建议：

- 复用现有 `SceneReferencePanel` 和 `ResultGallery` 中“设为场景参考图”的能力。
- 如果现有上传 API 只支持 `input/reference`，不要为了命名大改存储结构；UI 文案改为“新场景图”即可。

## 板块 2：分镜生成

职责合并：

- 上半区：原始分镜图上传、排序、创建/管理分镜组。
- 下半区：选择新场景图 + 分镜组，批量生成新分镜图。
- 显示原图和生成图对照。
- 支持下载分镜组生成结果。

命名：

- 标题：分镜生成
- 上半区：原始分镜图
- 下半区：批量生成
- 按钮：创建分镜组、开始分镜生成、下载 ZIP

重要原则：

- 不再使用“分镜重做”。
- 原始分镜图和生成操作不要拆成两个主板块。
- 如果没有新场景图，仍允许上传/整理分镜图；批量生成区提示“建议先在新场景图生成中选择一个新场景图，也可以稍后再选。”
- 如果没有分镜组，仍显示生成区，但提示“先从上方原始分镜图创建分镜组。”

实现建议：

- `ShotSetPanel` 可以重命名 UI 文案，但文件名不一定要立刻改，避免大范围 import 变更。
- 将 `openApplySceneModal` 或“批量应用场景”的交互整合到分镜生成板块内部，减少弹窗里的概念负担。
- 保留弹窗也可以，但按钮和弹窗标题要叫“开始分镜生成”。

## 板块 3：脚本生成

职责：

- 单独承载目标人群、语气、平台、卖点。
- 基于项目、分镜顺序、新场景图、卖点生成 15 秒口播脚本/视频文案。
- 显示历史脚本草稿。
- 支持复制、下载 txt/json。

命名：

- 标题：脚本生成
- 按钮：生成脚本

数据建议：

当前脚本接口从 `shot_sets.category` 里解析 sellingPoints，这是临时且语义不清晰。建议新增更清晰的持久化位置：

- 轻量方案：给 `projects` 增加 `targetAudience TEXT DEFAULT ''`、`tone TEXT DEFAULT '种草'`、`platform TEXT DEFAULT '通用'`、`sellingPointsJson TEXT DEFAULT '[]'`。
- 同步在 `lib/db.ts` migrations 中添加这些列。
- `app/api/projects/[id]/script/route.ts` 从 project 字段读取这些值。
- 如果暂时不做新列，至少不要继续把新卖点写入 `shot_sets.category`。

交互要求：

- 脚本生成可以在没有视频任务时使用。
- 如果没有分镜，提示“可以先写脚本；加入分镜后脚本会更具体。”
- 生成前允许用户编辑卖点，不需要回到新建项目页。

## 板块 4：视频生成

职责：

- 选择分镜组或单个分镜图。
- 选择视频供应商、运镜模板、时长。
- 使用脚本或自定义视频提示词。
- 创建视频任务、补抓结果、重试、下载。

命名：

- 标题：视频生成
- 按钮：生成视频、补抓结果、重试、下载

实现注意：

- 保留已修复的视频下载路径：`/api/videos/videos/${encodeURIComponent(job.filename)}`，因为视频实际保存到 `storage/videos/`。
- 视频生成不要和脚本生成放在同一个主板块。
- 如果没有脚本，允许用户输入自定义提示词生成视频。
- 如果没有分镜，板块内部提示“可以先在分镜生成中准备分镜图，也可以后续再回来。”

## 状态和提示设计

不要用硬性禁用主板块。每个板块内部用状态提示：

- 空状态：解释此板块能做什么。
- 缺材料提示：列出缺什么，不要拦住整个板块。
- 已完成提示：显示当前已选材料和下一步建议。
- 运行中提示：显示任务数量、成功、失败、进行中。

示例文案：

- “还没有新场景图。你可以上传原始场景图生成候选，也可以直接上传已有新场景图。”
- “已选择 1 张新场景图，可用于分镜生成。”
- “当前没有分镜组。先上传原始分镜图并创建分镜组。”
- “脚本可以先生成；加入分镜后，脚本会包含更具体的镜头顺序。”

## 推荐实现顺序

请 Claude Code 按这个顺序做，避免一次性大爆炸：

1. 供应商页：移除“设为唯一启用” UI 和相关前端逻辑。
2. 新建项目页：简化复杂产品项目表单，只创建项目壳。
3. 后端项目创建：放宽 complex workflow 的素材校验，支持空项目壳。
4. 项目详情页：重排为 4 个主板块，先保留现有组件能力。
5. 分镜生成：把“原始分镜图”和“批量生成”合并到同一主板块，替换文案。
6. 脚本生成：从项目详情中独立成主板块，并把卖点输入迁入这里。
7. 视频生成：独立成主板块，保留视频任务列表、补抓、重试、下载。
8. 统一中文文案，清理“场景图 B”“分镜重做”“设为唯一启用”等旧词。

## 验收标准

功能验收：

- 可以只填项目名、供应商和基础参数创建复杂产品项目。
- 新项目详情页能打开，且显示 4 个主板块。
- 用户可以在没有任何图片时进入每个板块，不出现整块灰掉或页面崩溃。
- 可以在“新场景图生成”里生成或选择新场景图。
- 可以在“分镜生成”里上传原始分镜图、创建分镜组、选择新场景图后批量生成。
- 可以在“脚本生成”里填写/修改卖点并生成脚本。
- 可以在“视频生成”里创建视频任务，并下载成功结果。
- 供应商页没有“设为唯一启用”。

代码验收：

- 不提交 `.next/`、`node_modules/`、`data/`、`storage/`。
- 保持 TypeScript 通过。
- `npm run lint` 没有 error；已有 warning 可暂时保留，但不要新增明显无用变量。
- `npm run build` 通过。

建议验证命令：

```powershell
npx tsc --noEmit --incremental false
npm run lint
npm run build
```

## Codex 后续审核重点

Claude Code 实现后，Codex 审核时重点看：

- 是否真的支持空项目壳创建，而不是只改了前端。
- 是否误删 legacy batch edit 能力。
- 是否把主步骤做成了硬锁或灰态。
- 是否还有“设为唯一启用”“场景图 B”“分镜重做”等旧文案漏网。
- 脚本卖点是否还写进 `shot_sets.category` 这种语义不清的位置。
- 视频下载路径是否保持正确。
- 构建产物和本地数据是否被误加入 git。
